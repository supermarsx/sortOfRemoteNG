# t5-e1 — backend-wire sorng-hetzner (76 commands)

## Goal
Wire all 76 `hetzner_*` Tauri commands from `sorng-hetzner` through the
`sorng-commands-cloud` flat handler, and register `HetznerService` as
managed state so the commands are reachable end-to-end.

## Root cause of prior "t3-e53 wired hetzner" false claim
`src-tauri/src/hetzner_commands.rs` wrapped `include!(...)` inside
`#[allow(dead_code)] mod inner { ... }` with **no** `pub(crate) use inner::*;`
re-export. Result: the `#[tauri::command]` macros compiled but all 76
fns stayed buried inside `inner`, unreachable by any `generate_handler!`.
Nobody was including the module either (`grep hetzner_commands` → 0 hits
outside its own file). So registered-command count was 0.

## Changes

1. **src-tauri/src/hetzner_commands.rs** — added the missing
   `pub(crate) use inner::*;` after the `mod inner { include!(...) }`
   block (matches `sftp_commands.rs` pattern). This file is still not
   mod'd anywhere in the app crate, but is now structurally correct if
   someone adds a `mod hetzner_commands;` declaration later.

2. **src-tauri/crates/sorng-commands-cloud/Cargo.toml** — added
   `sorng-hetzner = { path = "../sorng-hetzner" }`. (Needed because the
   `include!(...)` in the new shim reads the raw `commands.rs` source
   of `sorng-hetzner`; the dep also ensures crate presence in the build
   graph even though `hetzner` is already re-exported via
   `sorng-app-domains`.)

3. **src-tauri/crates/sorng-commands-cloud/src/hetzner_commands.rs**
   (new file) — shim mirroring the `gcp_commands.rs` pattern:
   ```rust
   mod service { pub use crate::hetzner::service::*; }
   mod types   { pub use crate::hetzner::types::*; }
   #[allow(dead_code)]
   mod inner { include!("../../sorng-hetzner/src/commands.rs"); }
   pub(crate) use inner::*;
   ```
   The `include!(...)` pulls the 76 `#[tauri::command]` fns from
   `sorng-hetzner` directly. Their `use super::service::HetznerServiceState;`
   and `use super::types::*;` resolve to the shim's neighbor modules.

4. **src-tauri/crates/sorng-commands-cloud/src/lib.rs** — added
   `mod hetzner_commands;` in the module list.

5. **src-tauri/crates/sorng-commands-cloud/src/cloud_handler.rs** —
   inserted all 76 command names into the flat `is_command` `matches!(...)`
   arms (before `smtp_add_profile`) and all 76 `hetzner_commands::*`
   entries into the `tauri::generate_handler!` list. Block is clearly
   delimited with the comment `// Hetzner commands (sorng-hetzner)`.

6. **src-tauri/src/state_registry.rs** — added
   `#[cfg(feature = "cloud")] pub(crate) use hetzner::service::HetznerService;`
   alongside the other cloud types.

7. **src-tauri/src/state_registry/security_data.rs** — added
   `let hetzner_service = HetznerService::new(); app.manage(hetzner_service);`
   alongside the other cloud services (GCP/Azure/Exchange/SMTP). Note
   that `HetznerService::new()` already returns
   `HetznerServiceState = Arc<Mutex<HetznerService>>`, which is exactly
   what the commands pull via `State<'_, HetznerServiceState>`.

## Verification

- `cargo check -p app --features cloud` — **GREEN** (9m37s,
  only pre-existing unused-import warnings; nothing new from t5-e1).
- `rg "hetzner_" src-tauri/crates/sorng-commands-cloud/src/cloud_handler.rs | wc -l`
  → **152** (76 `matches!` arms + 76 `generate_handler!` entries).
- All 76 commands now reachable end-to-end: frontend
  `invoke("hetzner_*")` → `invoke_handler::build()` → cfg-cloud branch
  → `sorng_commands_cloud::is_command` matches → `cloud_handler(invoke)`
  dispatches to the generated Tauri handler → command fn pulls
  `HetznerServiceState` from managed state (registered in
  `security_data.rs`) → `sorng-hetzner` service executes.
